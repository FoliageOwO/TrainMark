# TrainMark AI（智训批）项目实现计划

## 1. 项目定位

TrainMark AI，中文名“智训批”，是面向高校实训教学场景的“实训报告智能批改与管理系统”。系统覆盖学生提交、教师收集、AI 批改、人工复核、成绩发布、失分分析、查重检测、课程目标达成度分析和教学改进建议，目标是建设一个可以真实上线生产使用的完整产品，而不是仅满足演示的 MVP。

核心闭环：

```text
教师创建课程
→ 导入班级与学生名单
→ 创建实训任务
→ 配置评分标准与参考答案
→ 学生提交报告
→ 系统自动整理与校验
→ AI 识别、批改、生成批注
→ 教师复核与修正
→ 发布成绩与批注
→ 学生查看结果与申诉
→ 教师处理申诉
→ 导出成绩、失分分析、课程目标达成度报告
→ 沉淀评分模板、历史样本和教学改进数据
```

## 2. 产品目标

| 目标 | 说明 |
|---|---|
| 提升收集效率 | 学生在线提交，系统自动核对名单、归档文件、标记未交 |
| 降低批改负担 | 通过 OCR、语义分析、规则评分和 AI 评语生成完成初批 |
| 保证成绩可信 | 教师复核后发布，所有改分、发布、导出均留痕 |
| 支撑教学改进 | 自动统计高频失分点、班级差异、课程目标达成度 |
| 支持生产上线 | 完整权限、安全、审计、备份、监控、容错和部署方案 |

关键指标：

| 指标 | 目标 |
|---|---|
| 学生提交步骤 | 不超过 3 步 |
| 教师创建任务时间 | 5 分钟内完成常规任务配置 |
| 催交覆盖率 | 100% 覆盖未提交学生 |
| 单份报告 AI 批改耗时 | 平均不超过 30 秒 |
| AI 与人工评分误差 | 平均不超过 3 分 |
| 50 并发上传 | 服务稳定，无任务丢失 |
| 成绩统计生成 | 常规统计 3 秒内返回，复杂导出异步生成 |
| 审计日志保存 | 至少 1 年 |

## 3. 用户角色

| 角色 | 核心诉求 | 主要功能 |
|---|---|---|
| 学生 | 快速提交报告，查看成绩和批注 | 查看任务、上传报告、查看批注 PDF、查看扣分原因、提交申诉 |
| 教师 | 收集报告、批改、发布成绩、分析教学问题 | 创建课程任务、配置评分标准、AI 批改、人工复核、导出成绩、催交、失分分析 |
| 课程负责人 | 统一课程标准，查看多班级教学情况 | 管理课程模板、跨班统计、课程目标达成度分析 |
| 督导/系主任 | 查看教学质量和课程达成情况 | 查看批改进度、达成度报告、异常任务、成绩分布 |
| 系统管理员 | 保证系统稳定、安全、可维护 | 用户管理、组织管理、权限配置、系统配置、模型配置、日志审计、备份恢复 |

## 5. 总体架构

TrainMark AI 采用前后端一体化 Monorepo 管理。前端、后端、AI 服务、基础设施配置、部署脚本和文档全部放在同一个项目中，便于统一开发、测试和部署。

推荐目录结构：

```text
TrainMark/
├── apps/
│   ├── web/                         # React 前端，老师端、学生端、管理端
│   └── docs/                        # 产品文档、接口文档、部署文档
├── backend/
│   ├── gateway-service/             # 网关、统一鉴权、限流、路由
│   ├── auth-service/                # 登录、Token、权限认证
│   ├── user-service/                # 用户、组织、班级、角色
│   ├── course-service/              # 课程、教学班、任务、评分标准
│   ├── file-service/                # 文件上传、分片、预览、转换、存储
│   ├── grading-service/             # AI 批改、人工复核、批注生成
│   ├── ocr-service/                 # OCR 调用、版面结构识别
│   ├── similarity-service/          # 查重和相似度检测
│   ├── analytics-service/           # 成绩统计、失分分析、达成度
│   ├── notification-service/        # 邮件、企业微信、站内信
│   └── admin-service/               # 系统配置、审计、容量、日志
├── ai/
│   ├── paddleocr/                   # OCR 模型与配置
│   ├── scoring/                     # 评分模型、语义相似度、规则引擎
│   └── annotation/                  # PDF 批注生成服务
├── infra/
│   ├── docker-compose.yml           # 本地和测试环境编排
│   ├── nginx/                       # 反向代理配置
│   ├── minio/                       # 对象存储配置
│   ├── postgres/                    # 初始化 SQL 和迁移脚本
```

## 6. 技术选型

| 层级 | 技术 | 说明 |
|---|---|---|
| 前端 | React + TypeScript + Vite | 主应用开发 |
| UI | Ant Design Pro + 自定义主题 | 快速构建后台系统，同时保证精致 UI |
| 状态管理 | Zustand 或 Redux Toolkit | 管理用户、权限、任务、上传状态 |
| 数据请求 | Axios + OpenAPI 类型生成 | 保证前后端接口类型一致 |
| 后端 | Spring Boot 3 + Spring Cloud | 微服务架构 |
| 网关 | Spring Cloud Gateway | 统一鉴权、限流、路由 |
| 权限 | Spring Security + JWT + RBAC | 登录、角色、数据权限 |
| 数据库 | PostgreSQL | 事务数据、统计数据 |
| 缓存 | Redis | 会话、热点数据、分布式锁 |
| 队列 | RabbitMQ | OCR、批改、导出、通知异步处理 |
| 文件存储 | MinIO，生产可替换 S3/OSS | 原始报告、批注 PDF、导出文件 |
| OCR | PaddleOCR | 中文、表格、图片识别 |
| 语义模型 | SentenceTransformer / BGE 中文模型 | 语义相似度评分 |
| 大模型 | 本地模型或外部 API | 生成评语、改进建议、教学建议 |
| PDF 处理 | PDFBox / LibreOffice / 自定义批注层 | 转换、预览、批注生成 |
| 监控 | Prometheus + Grafana | 服务和任务监控 |
| 日志 | Loki 或 ELK | 日志检索、异常分析 |
| 部署 | Docker Compose 起步，生产可升级 Kubernetes | 环境一致性和扩展能力 |

## 7. 权限与数据隔离

权限模型：

```text
User
→ UserRole
→ Role
→ RolePermission
→ Permission
```

数据权限规则：

| 数据 | 规则 |
|---|---|
| 学生报告 | 学生只能访问自己的报告和批改结果 |
| 课程任务 | 教师只能访问授权课程和班级 |
| 成绩 | 未发布前学生不可见 |
| 批改记录 | 教师可见完整记录，学生只看最终结果 |
| 查重报告 | 默认仅教师、课程负责人和督导可见 |
| 审计日志 | 管理员和授权督导可见 |
| 文件下载 | 使用短期签名 URL，禁止永久公开直链 |

### 8.3 创建实训任务

创建任务采用分步向导：

```text
基础信息
→ 提交要求
→ 评分标准
→ AI 批改配置
→ 通知规则
→ 确认发布
```

基础信息：

| 字段 | 说明 |
|---|---|
| 任务名称 | 如“Java Web 综合实训报告” |
| 所属课程 | 选择课程 |
| 适用班级 | 多选班级 |
| 截止时间 | 支持精确到分钟 |
| 补交规则 | 允许或不允许，是否扣分 |
| 总分 | 默认 100 |
| 是否查重 | 开启或关闭 |
| 是否 AI 自动批改 | 开启或关闭 |

提交要求：

| 配置 | 说明 |
|---|---|
| 文件格式 | PDF、Word、JPG、PNG |
| 文件大小 | 默认不超过 50MB |
| 文件数量 | 单文件或多文件 |
| 命名规则 | 系统自动统一命名，不依赖学生文件名 |
| 封面校验 | 是否要求包含姓名、学号、课程 |
| 重复提交 | 截止前允许覆盖，保留历史版本 |
| 提交说明 | 老师填写任务要求和注意事项 |

评分标准：

| 配置 | 说明 |
|---|---|
| 评分项 | 如需求分析 20 分、系统设计 25 分、实现结果 30 分 |
| 得分点 | 每个评分项下配置多个得分点 |
| 关键词 | 必须出现或建议出现 |
| 同义词 | 如“数据库连接池”和“HikariCP” |
| 步骤分 | 支持按步骤判分 |
| 扣分规则 | 缺图、缺表、格式错误、截图不清晰 |
| 课程目标 | 每项评分关联 CO1、CO2、CO3 |

## 14. AI 批改方案

TrainMark AI（智训批）的批改能力不应依赖单一大模型直接打分，而应采用可解释、可复核的组合式评分流水线。

流水线：

```text
文件预处理
→ PDF/Word/图片统一转换
→ OCR 识别
→ 版面分析
→ 标题、段落、表格、图片、代码块提取
→ 与评分标准匹配
→ 关键词命中
→ 语义相似度计算
→ 规则扣分
→ 大模型生成评语
→ 置信度计算
→ 批注 PDF 生成
→ 教师复核
```

评分策略：

| 类型 | 默认权重 |
|---|---|
| 关键词匹配 | 30% |
| 语义相似度 | 40% |
| 结构完整性 | 15% |
| 格式规范 | 10% |
| 教师自定义规则 | 5% |

置信度分级：

| 置信度 | 处理方式 |
|---|---|
| 90% 及以上 | 可批量通过 |
| 70%-90% | 正常复核 |
| 50%-70% | 优先复核 |
| 50% 以下 | 标记异常，建议人工批改 |

低置信度原因：

| 原因 | 说明 |
|---|---|
| OCR 质量低 | 图片模糊、扫描倾斜、文字过小 |
| 文档结构异常 | 缺页、缺章节、空白页过多 |
| 语义边界模糊 | 与多个评分档位相近 |
| 查重风险高 | 与其他报告高度相似 |
| 规则冲突 | 多条评分规则给出不同判断 |

```text
某课程目标达成度 = 该目标关联评分项学生平均得分 / 该目标关联评分项满分
```

| 文件 | 内容 |
|---|---|
| 成绩登记表.xlsx | 姓名、学号、班级、各项分、总分、排名 |
| 批改明细表.xlsx | 每个评分项得分和扣分原因 |
| 失分分析.pdf | 高错点、分布图、教学建议 |
| 课程目标达成度.pdf | CO 达成度、雷达图、支撑数据 |

## 19. 安全与合规

| 项目 | 措施 |
|---|---|
| 登录安全 | JWT 短期 Token + Refresh Token |
| 密码安全 | BCrypt 加密 |
| 权限控制 | RBAC + 数据权限 |
| 文件安全 | 白名单格式、病毒扫描、大小限制 |
| 传输安全 | HTTPS |
| 存储安全 | 私有桶、签名 URL |
| 防刷限流 | 网关按用户、IP、接口限流 |
| 防越权 | 所有接口校验用户数据范围 |
| 审计 | 分数修改、导出、删除、发布全部留痕 |
| 数据备份 | 每日全量 + 每小时增量 |
| 日志脱敏 | 学号、手机号、邮箱脱敏显示 |

必须避免的问题：

| 风险 | 处理 |
|---|---|
| 学生访问他人报告 | 服务端强制按 userId 过滤 |
| 教师访问未授权课程 | 按 course_member 校验 |
| 成绩未发布提前泄露 | 发布状态控制 |
| 文件直链泄露 | 使用短期签名 URL |
| 大文件拖垮服务 | 分片上传 + 异步处理 |
| AI 改分无依据 | 保留规则版本、模型版本和评分日志 |

## 20. 性能与可靠性

性能目标：

| 场景 | 目标 |
|---|---|
| 登录 | 小于 1 秒 |
| 课程列表 | 小于 1 秒 |
| 提交列表 | 小于 1.5 秒 |

| 常规统计报表 | 小于 10 秒，复杂报表异步生成 |
| 50MB 文件上传 | 稳定上传，支持失败重试 |

优化策略：

| 策略 | 说明 |
|---|---|
| 分片上传 | 避免大文件失败重传 |
| 异步队列 | OCR、批改、导出、通知均走 RabbitMQ |
| Redis 缓存 | 缓存课程、权限、统计摘要 |
| 数据库索引 | assignment_id、student_id、status、created_at |
| 批量任务 | 批量批改分批消费 |
| 限流 | 上传、导出、批改接口限流 |
| 预生成统计 | 批改完成后异步生成分析数据 |

可靠性设计：

| 项目 | 设计 |
|---|---|
| 健康检查 | 每个服务暴露 `/actuator/health` |
| 任务重试 | OCR、批改失败自动重试 3 次 |
| 死信队列 | 重试失败进入人工处理队列 |
| 幂等处理 | 上传完成、批改任务、成绩发布均保证幂等 |
| 文件校验 | MD5 校验，避免损坏 |
| 数据备份 | PostgreSQL 和 MinIO 定期备份 |
| 灾备目标 | RPO 不超过 1 小时，RTO 不超过 4 小时 |
| 链路追踪 | traceId 串联一次批改全过程 |

## 28. 最终建议

TrainMark AI（智训批）不应被实现为“上传文件后 AI 给一个分数”的简单演示系统，而应建设为教育场景下可信、可解释、可复核、可追溯的智能批改与教学管理平台。

最重要的产品抓手：

| 抓手 | 原因 |
|---|---|
| 教师复核工作台 | 决定教师是否敢用、愿意用 |
| 评分标准编辑器 | 决定 AI 批改是否可解释、可控 |
| 失分分析与达成度 | 决定系统是否真正服务教学改进 |

最重要的工程抓手：

| 抓手 | 原因 |
|---|---|
| 异步批改队列 | 决定系统稳定性和可扩展性 |
| 文件与权限安全 | 决定系统能否真实上线 |
| 审计与可追溯 | 决定成绩场景能否被信任 |

建议落地顺序：

1. 搭建 Monorepo、前端布局、登录权限、数据库基础表。
2. 实现老师端课程任务与学生端提交闭环。
3. 实现评分标准编辑器和 AI 批改流水线。
4. 实现人工复核、成绩发布、学生查看结果。
5. 补齐统计分析、查重、管理端、生产部署和监控告警。
